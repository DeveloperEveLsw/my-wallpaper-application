# ADR 0001: WPF 네이티브 애플리케이션을 제품 런타임으로 사용

- 상태: 채택
- 날짜: 2026-07-19

## 배경

제품은 사용자가 선택한 실제 Windows 폴더와 파일을 읽고, 네이티브 클릭·우클릭·드래그
입력을 처리하며, 파일 이동을 실제 파일 시스템에 반영해야 한다. 웹 월페이퍼를 제품
본체로 사용하면 브라우저 샌드박스, 호스트 입력 전달과 파일 시스템 접근 경계가 핵심
기능을 제한할 수 있다.

## 결정

제품 본체는 C#/.NET 10 LTS 기반 WPF 애플리케이션으로 구현한다.

Wallpaper Engine과 Lively는 WPF 프로세스를 실행하고 바탕화면에 배치하며 생명주기를
제어하는 선택 가능한 호스트로만 사용한다. 파일 시스템, 상호작용 상태와 제품 UI는
WPF 프로세스가 소유한다. 개발과 진단을 위한 Standalone 실행 모드도 동일한 기능을
제공한다.

Wallpaper Engine 통합은 개인 로컬 Application Wallpaper로 한정한다. 공개 Workshop
배포는 지원하지 않는다. Wallpaper Engine/Lively 속성에 제품 설정을 연결하지 않고
WPF 자체 설정 UI와 앱 로컬 설정을 사용한다.

## 결과

장점:

- .NET과 Windows Shell API를 통한 직접적인 파일 시스템 통합
- 브라우저 샌드박스에 의존하지 않는 입력과 컨텍스트 메뉴
- WPF 데이터 바인딩, 스타일, 애니메이션을 활용한 Glass UI
- 특정 월페이퍼 호스트 장애와 제한을 어댑터 내부로 격리
- 호스트 없이도 애플리케이션 기능을 개발하고 검증 가능

비용과 위험:

- WPF 실행 및 실제 UI 검증은 Windows에서만 가능
- 호스트별 HWND 재배치, 입력, pause/resume 계약을 각각 검증해야 함
- DWM, Explorer와 월페이퍼 호스트의 창 계층 차이를 실환경에서 다뤄야 함
- WSL 자동 검증만으로 완전한 완료 판정을 내릴 수 없음

## 기각한 대안

- Web wallpaper 중심: 핵심 파일·입력 기능이 웹 및 호스트 경계에 종속된다.
- Electron 중심: 웹 기반 렌더러와 네이티브 브리지 계층을 다시 필요로 하며 WPF보다
  현재 요구에 직접적이지 않다.
- 특정 호스트 전용 플러그인: 호스트 교체와 Standalone 검증이 어려워진다.
